Introduction

This section should provide an brief overview of the entire document.

### 1.1 Purpose

The purpose of this document is to present the architectural design and analysis model for the SCEMAS
system. It outlines the system’s structure, including its subsystems, architectural styles, and class responsibilities.
The document is intended for internal stakeholders, such as project managers, developers, and system designers, involved in the development and implementation of the SCEMAS platform. The prior deliverable,
including the Software Requirements Specification (SRS), should be read beforehand to provide additional
context for the system requirements.

### 1.2 System Description

SCEMAS (Smart City Environmental Monitoring and Alert System) is designed to collect, process, and
distribute environmental sensor data across a city. The system ingests telemetry data from IoT sensors,
validates and stores the data, and makes it available to operators and the public through dashboards and
APIs. The platform also includes an alert management system that allows operators to define environmental
thresholds and receive notifications when abnormal conditions occur, along with access control mechanisms
for managing user accounts, permissions, and device authentication.
The system follows a Presentation-Abstraction-Control (PAC) style architecture, with several subsystems
using additional architectural patterns suited to their specific responsibilities. The telemetry ingestion subsystem follows a pipe-and-filter architecture to process sensor data through a sequence of validation and
transformation stages. The access management subsystem follows a repository-style architecture, providing
a centralized passive data store for managing accounts, permissions, and authentication data. The alert
management subsystem follows a blackboard architecture, enabling different components to react to alert
conditions and update alert states collaboratively.

### 1.3 Overview

The remainder of this document describes the architectural structure and design of the SCEMAS system.
Section 2 presents the Analysis Class Diagram, illustrating the key classes and their relationships within

subsystem structure, and the reasoning behind key design decisions. Finally, Section 4 presents the Class

interactions within the system.

### 3.1 System Architecture

SCEMAS follows an Interaction-Oriented Software Architecture, more specifically best described using the
PAC architecture, where the system is decomposed into a set of co-operating agents. The presentation
represents what the user sees and interacts with, being the dashboard views. Abstraction holds the key
data and domain logic supported by each agent. Control coordinates the logic between each presentation
and abstraction, as well as mediating communication between higher and lower-level agents to compose the
system hierarchically.
Within PAC architecture, it can be broken down into various agents with differing architectural styles.

PAC was selected as the global architecture for SCEMAS due to a variety of substantial reasons. SCEMAS
serves three fundamentally different user roles; City Operator, Public Users, and System Administrators,
each requiring a different view of a specific subset of the underlying data. The overall composition supports
various simultaneous agents mapping different views with their own independent Abstraction to encourage
decision making. For the operator’s dashboard, key alert decisions are based on the data being represented
in a cohesive way specific to them, where they should not be concerned with other views that represent
unrelated data. Unlike a shared-model architecture like MVC that forces all three views to share a single
domain model, PAC naturally isolates these interaction models while still allowing coordination through
control components ensuring loose coupling.
SCEMAS is inherently data driven, including telemetry updates, rule changes, and alert triggers. Although
broken into subsystems that represent different architectures, PAC will support multiple processes of interaction via a root-level Data Distribution Management Controller. Telemetry Management and Alerting
Management get data from a shared entity called Individual Sensor Reading. This data from the shared
entity is used by the Alerting Management Controller for decision support of whether to trigger any alert or
not. Finally, the Data Distribution Management Controller distributes or presents views to the dashboard,
which surfaces alerts to operators who acknowledge them. Each agent operates independently, communicating through shared entities rather than direct control-to-control communication. This cleanly separates
concerns between agents, ensuring each is only responsible for its own tasks without needing to understand
the implementation of other agents. Modularity is also another important feature to SCEMAS to support
various dashboard analytics, and/or alert management. PAC ensures new agents can be added to the tree
without affecting the existing agents supporting the Open-Closed Principle where additional features or
elements will not overwrite existing code but supplement the overall system.

In terms of our sub-system architectural design, we broke our system into three parts; Telemetry Management, Access Management, and Alerting Management. Each has different architectures but corresponds to
the overall PAC architecture to communicate.
Telemetry Management although showing Blackboard-like characteristics with an active data store notifying
subsequent agents with data changes, the premise of the architecture is not met; independent knowledge
sources are not working simultaneously to provide a solution to a non-deterministic problem. Telemetry
Management only represents a sequential transformation pipeline to ingest into various sources that handle
their specific logic. Raw MQTT message → Schema Validation → Value Range Validation → Timestamp
Validation → Persistence → Aggregation. Each stage takes the output of the previous stage as input,
transforms and processes it, and passes the resulting structured data packet to the next filter through a
pipe, without knowledge of the stages upstream or downstream. This clearly represents the Pipe and Filter
architectural style.

Access Management, on the other hand, provides a single authoritative source of truth, with a simple
transactional process. User credentials, and RBAC roles are not event-driven or non-deterministic; they are
queried deterministically on each authentication request. Authentication requests are clearly initiated by
active agents, querying a passive data store that deterministically returns the corresponding role.
Alerting Management clearly represents a blackboard architecture. At its core, it’s reactive, data-driven
and requires certain knowledge sources to become fulfilled. The lifecycle of an alert, triggered → active →
acknowledged → resolved, is a shared data state that multiple consumers react to without being directly
coupled to one another. Independent knowledge sources are agents that react to new telemetry, categorize
it based on specific thresholds exceeded or anomalous trends, and post it for new rule definitions, and
acknowledgement. With alerts being a non-deterministic problem, each agent works to confidently classify
the alert event.
Naturally, with a dashboard view, MVC was the first design choice to consider. However, it was shortly
eliminated as MVC assumes a single shared model synchronized across all views. With at least three different
views; City Operators, Administrators, and Public Users, each serve fundamentally different services with
different subsets of data. Meaning a shared model would become a uniform object carrying all user contexts
simultaneously, creating high coupling between unrelated concerns.
Batch Sequential was also eliminated as SCEMAS has various latency requirements. For instance, operators
must be able to acknowledge alerts within 30 seconds. Batch Sequential processes records in staged batches
with no interactive feedback between stages, which is fundamentally incompatible with a live monitoring
dashboard that needs real-time alert responses.
Process Control was eliminated easily as it is typically used for the control of an embedded system with
continuing physical actions. With this system, none of the observational outputs represented in the dashboard
feedback into a measured variable, meaning the closed feedback loop that defines a Process-Control is never
formed.
Data-centered architectures, specifically Repository and Blackboard, were considered but eliminated as the
overall system architecture. Applying Repository globally would model SCEMAS as a passive data store,
providing no inherent mechanism for the interactive, role-differentiated user experiences the system requires.
There is no means to express hierarchical agent composition or per-user view differentiation, an operator’s
personalized alert subscription state must remain independent from another operator’s session, which a
globally shared repository cannot cleanly represent. Similarly, modelling the entire system as a Blackboard
would require all components, including structured account management and role-gated user interfaces to
be expressed as knowledge sources reacting opportunistically to a shared data space, which is inappropriate
for the deterministic, transactional nature of authentication and RBAC. Both styles are retained at the
subsystem level where their properties are directly applicable, but neither is sufficient to describe the system.

### 3.2 Subsystems

Our main system has been subdivided into three distinct subsystems: Telemetry Management, Alerting
Management, and Access Management.
The Telemetry Management subsystem is responsible for all real time collection and ingestion of data,
dealing with individual sensor readings, validating its data, monitoring ingestion health and maintaining an
up-to-date database of all telemetry related information. This subsystem depends on Access Management
to authenticate each incoming IoT device and abides by the pipe and filter architecture style. Telemetry
Management and Alerting Management operate as PAC agents under the overall PAC architecture, with
Access Management acting as a supporting Repository subsystem.
The Access Management subsystem provides user capabilities such as signing up, logging in, modifying
account information, as well as monitoring all security details, including device identity, security permissions,

access status, being the subsystem to maintain the account database, IoT device identity registry and security
data. This subsystem does not communicate with any of the others, instead acting as a dependency for both
Telemetry Management and Data Distribution Management, which depend on it to verify the identity of
IoT devices and authenticated users respectively.
The Alerting Management subsystem defines all functionalities regarding the SCEMAS rule-based alerting
system, allowing for definition of rules, managing and distributing of alert notifications, audit logs of all alert
events, and the overall management of the alert and audit log database. This subsystem retrieves subscriber
information from the shared User Information entity found in the Account Database to determine who to
notify upon a threshold violation.
